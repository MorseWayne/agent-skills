# Go 后端设计文档 Review (Design Docs Only)

本 Skill 用于对 Go 后端项目的**需求与设计文档**进行深度审查。

**核心原则**：

1. **只看设计，不看代码**：仅基于 Markdown 文档、接口定义 (.proto/Swagger) 和数据库脚本 (.sql) 进行评估。
2. **一致性优先**：确保“功能 A”在 README、需求规格 (SPEC) 和技术设计 (Design) 中的描述是完全对齐的，没有歧义或冲突。

## 适用场景

- **需求评审**：检查需求文档 (SPEC) 是否清晰、完整。
- **设计评审**：检查技术方案 (Design) 是否覆盖了所有需求，且符合工业化标准。
- **文档一致性治理**：检查项目文档是否存在陈旧、冲突或描述不当的问题。

## Review 流程

### 第一步：设计资料收集 (Docs Only)

Agent 仅读取以下设计类文件，**严禁读取任何 .go 源代码文件**：

1. **顶层文档**：
    - `{workspace}/README.md` (项目入口、核心价值、快速开始)
    - `{workspace}/docs/**` (需求规格 SPEC.md、技术设计 design.md、迁移指南等)

2. **接口契约** (作为设计的一部分)：
    - `{workspace}/api/**` (.proto 文件或 OpenAPI/Swagger 定义)
    - *注意：不读取 pkg/api 下的 Go 文件*

3. **数据契约** (作为设计的一部分)：
    - `{workspace}/scripts/sql/**` (DDL 建表语句)
    - *注意：不读取 internal/model 下的 Go 文件*

**执行指令**：
> "请扫描上述路径下的 Markdown、Proto 和 SQL 文件。不要读取任何 Go 源代码。"

### 第二步：深度一致性检查 (Deep Consistency)

这是本 Skill 的核心。必须交叉比对不同文档，找出描述冲突的地方。

#### 2.1 功能描述一致性 (Functional Alignment)

对比 README (概览)、SPEC (需求) 和 Design (实现方案)：

- [ ] **功能范围**：README 中宣称的功能，在 SPEC 中是否有详细需求？在 Design 中是否有对应的实现方案？
- [ ] **逻辑一致**：SPEC 中描述的业务流程（如状态流转），与 Design 中的时序图/流程图是否完全一致？
- [ ] **约束条件**：不同文档中对限制条件（如最大并发数、支持的字符集、超时时间）的描述是否冲突？

#### 2.2 接口契约一致性 (Interface Alignment)

对比 Design 文档与 .proto/Swagger 定义：

- [ ] **字段对齐**：Design 文档中提到的关键字段，是否都已在 .proto 文件中定义？类型是否一致？
- [ ] **枚举一致**：Design 文档中列举的状态/类型值，与 .proto 中的 enum 定义是否一致？

#### 2.3 数据模型一致性 (Data Alignment)

对比 Design 文档与 .sql 建表语句：

- [ ] **表结构**：Design 文档中的 ER 图或字段表格，与 .sql 文件的实际建表语句是否一致？
- [ ] **索引设计**：Design 文档中提到的查询性能要求，在 .sql 中是否有对应的索引设计支持？

### 第三步：工业化设计质量检查 (Quality by Design)

检查**设计文档本身**是否体现了高质量的软件工程思维。

#### 3.1 职责与边界

- [ ] **模块边界**：设计文档是否清晰定义了各模块的职责？是否存在职责模糊地带？
- [ ] **外部依赖**：是否明确列出了所有外部依赖（数据库、第三方服务、中间件），并说明了依赖理由？

#### 3.2 接口设计规范

- [ ] **规范性**：接口定义是否符合 RESTful 或 gRPC 最佳实践（基于文档描述）？
- [ ] **兼容性**：文档是否说明了接口的版本控制策略？是否考虑了向后兼容？

#### 3.3 非功能性设计 (Non-functional)

- [ ] **容错设计**：文档是否显式规划了重试、熔断、降级策略？
- [ ] **可观测性**：文档是否规定了关键路径需要输出哪些 Log、Metrics 和 Tracing 信息？
- [ ] **安全性**：文档是否描述了鉴权模型、敏感数据处理方式？

#### 3.4 异常处理

- [ ] **错误码**：文档是否定义了统一的错误码规范？
- [ ] **边缘情况**：SPEC 或 Design 中是否讨论了异常场景（如网络超时、数据冲突）的处理流程？

### 第四步：生成 Review 报告

生成一份专注于“文档质量与一致性”的报告：

```markdown
# 设计文档 Review 报告：{项目名称}

## 1. 概览
- **Review 范围**：[列出读取的文档]
- **一致性评级**：[高 / 中 / 低] - *反映各文档描述是否冲突*
- **完整性评级**：[高 / 中 / 低] - *反映设计是否覆盖所有需求*

## 2. 一致性冲突 (Consistency Conflicts) 🔴
> 重点列出不同文档对同一功能描述不一致的地方。
- **冲突点 1**：{功能名称}
  - **文档 A 描述**：... ({路径})
  - **文档 B 描述**：... ({路径})
  - **分析**：两者逻辑冲突/参数不一致...

## 3. 设计缺失 (Missing Designs) 🟡
> 只有需求没有设计，或只有概览没有细节。
- **缺失点 1**：README 提到了支持 X 功能，但 SPEC/Design 中未找到详细说明。
- **缺失点 2**：Design 中提到了数据库存储，但未提供 Schema 设计或 SQL 脚本。

## 4. 质量建议 (Quality Suggestions)
- **接口设计**：建议在 Design 文档中补充 API 的错误码定义。
- **非功能性**：SPEC 中缺少对高并发场景下的性能约束描述。

## 5. 总结
[一句话评价文档质量，重点指出是否可以进入开发阶段]
```

## 最佳实践检查清单 (Doc Checklist)

在 Review 文档时，请关注以下细节：

1. **单一事实来源 (SSOT)**：是否存在多个文档重复描述同一细节且容易导致不同步？建议通过引用解决。
2. **过时信息**：是否有文档显式标记了“最后更新时间”却很久未更新？
3. **模糊用词**：是否存在“可能”、“大概”、“后续考虑”等模糊词汇掩盖了设计决策？
4. **图文一致**：文档中的架构图/时序图是否与文字描述一致？
5. **示例准确性**：文档提供的 Request/Response JSON 示例是否符合字段定义？
